# 체험단 방문 등록 (마케터용 페이지)

인하우스 마케터가 무가(체험단) 방문을 직접 등록하면, 그 데이터가 **방문관리 페이지(`visitors`)에 즉시 반영**됩니다.
수작업 전산 입력 없이, 시차와 누락을 없애는 게 목적입니다.

## 동작 방식

1. 마케터가 로그인 → 양식에 입력 → 등록
2. 등록과 동시에 `visitors` 컬렉션에 문서가 추가됨 → **방문관리 페이지에 바로 표시**
3. 마케터는 **자기가 등록한 것만** 목록에서 보임
4. 관리자(나)는 **전체 마케터의 등록을 한 화면에서** 봄 (등록자 이메일도 표시)

## 파일 구성

| 파일 | 역할 |
|------|------|
| `marketer.html` | 화면 |
| `marketer.css` | 스타일 (밝은 톤) |
| `marketer.js` | 로직 — 로그인, 권한 분기, visitors 등록 |
| `firebase-config.js` | Firebase 설정 |
| `firestore.rules` | **통합 보안 규칙** (visitors/seeding/tasks/admins) |

---

## 설치 순서

### 1. 보안 규칙 게시 (가장 중요)
Firebase 콘솔 → Firestore Database → **규칙** 탭 → `firestore.rules` 내용 붙여넣고 **게시**.
이 규칙 하나가 시딩·방문관리·마케터 페이지 전부를 커버합니다.

> 기존에 `tasks` 컬렉션을 쓰고 있다면 규칙에 이미 포함해 두었습니다(로그인 사용자 허용).
> 권한을 다르게 쓰고 있었다면 그 부분만 조정하세요.

### 2. 관리자(나) 등록
완벽한 권한 분리를 위해 **admins 컬렉션** 방식을 씁니다.

1. Firebase 콘솔 → **Authentication** → 사용자 목록에서 **내 계정의 사용자 UID**를 복사
   (UID는 `abc123XYZ...` 형태의 긴 문자열)
2. Firestore → `admins` 컬렉션 시작 → 문서 추가
   - **문서 ID = 복사한 내 UID**
   - 필드는 비워도 되고, 원하면 `role: "admin"` 하나 넣어도 됨
3. 끝. 이 계정으로 로그인하면 마케터 페이지에서 "관리자" 배지가 뜨고 전체가 보입니다.

> 관리자를 더 추가하려면 그 사람 UID로 admins 문서를 하나 더 만들면 됩니다.

### 3. 마케터 계정 만들기
Firebase 콘솔 → Authentication → **사용자 추가**로 마케터별 이메일/비밀번호 생성.
마케터에게는 그 계정만 알려주면 됩니다. admins에 넣지 않았으니 자기 것만 보입니다.

### 4. 호스팅
다섯 파일(`marketer.html`, `marketer.css`, `marketer.js`, `firebase-config.js`)을
같은 폴더에 올립니다. 예: 기존 사이트의 `/marketer/` 경로.
반드시 `http(s)://`로 접속 (file://로 직접 열면 인증이 안 됩니다).

## visitors 문서 필드 (방문관리 페이지와 동일)

마케터 등록 시 아래 필드가 자동으로 채워집니다. 기존 방문관리 페이지가 그대로 읽습니다.

| 필드 | 채워지는 값 |
|------|-------------|
| id_name | 업체명(호점) |
| sns | 인플루언서 SNS 링크 |
| date | 방문 예정일 (YYYY-MM-DD) |
| time | 방문 시간 (없으면 "미정") |
| gift | 마스크팩(제품) |
| gift_ds | 디에스 포함 여부 (기본 true) |
| gift_others | 비고 |
| image | "" (빈 값) |
| order | 자동 증가 번호 |
| **ownerUid** | 등록한 마케터 UID (격리용, 추가됨) |
| **ownerEmail** | 등록한 마케터 이메일 (관리자 화면 표시용, 추가됨) |
| createdAt | 서버 시각 |

`ownerUid`, `ownerEmail`은 방문관리 페이지가 안 쓰던 새 필드라, 추가돼도 기존 화면에 영향 없습니다.

## 마케터 간 격리 수준 (꼭 읽어보세요)

지금 방문관리 페이지는 **공개 읽기**입니다(로그인 없이 열림). 이 선택 때문에 한 가지 한계가 있습니다:

- 마케터 페이지 화면에서는 자기 것만 보입니다(쿼리로 본인 ownerUid만 조회).
- 하지만 `visitors`가 공개라, 기술에 밝은 사람이 개발자도구로 마음먹으면 전체 데이터를 들여다볼 여지가 원칙적으로 남습니다. = **"화면 수준 격리"**입니다.
- 일반 업무 환경에서는 충분하지만, **완벽한 차단은 아닙니다.**

### 완벽 격리가 필요해지면
방문관리 페이지에 로그인을 붙이고, `firestore.rules`의 visitors 읽기 규칙을
`allow read: if true;` → `allow read: if isAdmin() || resource.data.ownerUid == request.auth.uid;`
로 바꾸면 서버 차원에서 완전히 막힙니다. (지금은 ownerUid를 미리 심어두었으니 전환만 하면 됩니다.)

---

## 시딩 페이지와의 관계
- 국내 제품 시딩 보드(`seeding` 컬렉션): 별개. 관리자용, 로그인 필수.
- 이 마케터 페이지(`visitors` 컬렉션): 체험단 방문, 방문관리와 연동.
- 둘은 같은 Firebase 프로젝트를 쓰지만 컬렉션이 분리돼 서로 영향 없습니다.
